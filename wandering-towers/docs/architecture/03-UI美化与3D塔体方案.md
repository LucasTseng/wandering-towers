## 1. 背景与目标

当前前端（`packages/client/src`）为纯平面俯视角：

- 棋盘 [Board.tsx](../../packages/client/src/components/Board.tsx)：16 空间环形，`radius=200`，每格 `60×60px` 绝对定位。
- 塔 [Tower.tsx](../../packages/client/src/components/Tower.tsx)：`TowerBlock` 平面 `36×20px` 棕色矩形 + 🏰/🛡 emoji。
- 巫师 [Wizard.tsx](../../packages/client/src/components/Wizard.tsx)：`WizardPiece` 纯色圆点 `14–20px`。
- 布局 [GameContainer.tsx](../../packages/client/src/containers/GameContainer.tsx)：`200px 1fr` 网格（左玩家信息 / 右棋盘），底部 `1fr 1fr 280px`（手牌/法术/日志）。

**本次目标**

1. 把平面图形标记升级为 **3D 模型**，重点强化塔的样式与「巫师立于塔上」的状态。
2. **严格依据规则 §10.5（塔顶最多 6 名可见巫师）反推塔顶平台尺寸**，保证 6 名巫师在视觉上确可放下。
3. 经尺寸推算，巫师在默认全景下偏小 → 把「游戏舞台区（地图+塔+巫师）」剥离为**独立可缩放/可平移区域**，支持滚轮缩放、拖放平移、一键还原。
4. 沉淀本增强的设计文档。

| 约束 | 值 | 来源 |
|---|---|---|

## 3. 尺寸推算（核心）

采用统一世界单位 **su（stage unit）**，渲染时按缩放系数映射为像素。

### 3.1 由「6 巫师」反推塔顶平台

设巫师底面直径 `D_w = 1.0 su`（最小可读单位）。6 名按 **3 列 × 2 行**排布于塔顶平台：

- 宽 = `3·D_w + 2·gap` = `3 + 2×0.35` = **3.7 su**
- 深 = `2·D_w + 1·gap` = `2 + 0.35` = **2.35 su**
- 取整含余量 ⇒ **塔顶平台 = 4.0 × 4.0 su**（6 名宽松可放，平台呈方形便于建模）

### 3.2 逐级反推（世界单位）

| 元素 | 尺寸 (su) | 依据 |
|---|---|---|
| 巫师底面直径 | 1.0 | 最小可读单位 |
| 塔顶平台 | 4.0 × 4.0 | 容纳 6 巫师（3×2） |
| 塔体（单段） | 顶 4.0×4.0 / 底 4.4×4.4 / 高 2.5 | 微张棱台，块状比例 ≈0.6 |
| 空间格 | 7.0 × 7.0 | 塔基 4.4 + 周围地面巫师位 |
| 环上格距 pitch | 7.5 | 7.0 格 + 0.5 缝 |
| 环半径（格心） | 19.1 | `16 × 7.5 ÷ 2π` |
| 环外径（含格深） | ≈ 22.6 | 19.1 + 3.5 |
| 舞台外接边长 | ≈ 50 | 含边距 |

### 3.3 像素映射

设 fit-all（全景默认）下 `1 su = 18 px`（舞台 ≈ 900 px，适配桌面主区）：

| 元素 | @ fit-all (18px/su) | @ 2× 缩放 | @ 4× 缩放 |
|---|---|---|---|
| 巫师 | 18 px | 36 px | 72 px |
| 塔顶平台 | 72×72 | 144×144 | 288×288 |
| 单塔高 | 45 px | 90 px | 180 px |
| 5 塔堆 | 225 px | 450 px | 900 px |
| 环外径 | ≈ 407 px | 814 px | 1628 px |
| 舞台外接 | ≈ 900 px | 1800 px | 3600 px |

### 3.4 结论（为何必须独立可缩放舞台）

- fit-all 下巫师仅 **18 px**（与现状 14–20 px 相当），3D 塔体堆叠却很醒目（5 塔 ≈ 225 px）。
- 要让 3D 巫师模型可读、便于点击，需 **2× 缩放（巫师 36 px）**；此时舞台 1800 px 超出视口 → 必须平移。
- ⇒ **舞台区独立 + 滚轮缩放 + 拖放平移 + 一键还原** 是刚需，与需求判断一致。

### 3.5 堆叠投影校核

极端 9 塔全堆一 格 = `9×2.5 = 22.5 su`。在斜俯相机（`rotateX 55°`，投影系数 `cos55°≈0.57`）下屏高 ≈ 12.8 su；相邻格距 7.5 su。极端高堆会视觉遮邻格，但正常 2–4 塔堆投影 2.9–5.7 su，远小于 7.5 su 间距，无遮挡。极端情形由缩放/平移补救，可接受。

## 4. 渲染技术选型

### 4.1 推荐：CSS 3D Transforms（方案 A）

- 每段塔 = `transform-style: preserve-3d` 的 div，由 4 侧面 + 1 顶面拼成棱台；顶面即塔顶平台。
- 巫师 = 极简 3D 棋子（尖帽锥 + 袍身），2–3 个 div 拼合，或贴图 billboard。
- 舞台置于 `perspective` 容器，相机以 `rotateX(55deg)` 给出斜俯视角。
- 缩放/平移：外层 `transform: translate(panX,panY) scale(zoom)`。
- **交互保持 DOM 点击**（巫师/塔/空间仍是真实元素）——无需 raycast，沿用现有 `onWizardClick/onTowerClick/onSpaceClick`。
- 零新依赖，契合现有 React+Vite 无重包姿态。

成本：9 塔 × 5 面 × 16 格 + 巫师 ≈ 数百 DOM 节点，React 可承受。材质/阴影受限于 CSS（渐变 + box-shadow + filter 可达「石质+烛光」质感）。

### 4.2 备选：three.js / react-three-fiber（方案 B）

- 真 3D、光照阴影佳、可上 GLTF 模型；后续法术 VFX 更顺。
- 成本：新增依赖（≈600 KB）、需 raycast 重建交互层、Board/Space/Tower/Wizard 大改。

### 4.3 决策

**本次采方案 A（CSS 3D）**：保留点击驱动交互（游戏 UI 的核心）、零依赖、落地快。方案 B 列为未来升级路径（当法术动画/粒子需求增长时再迁）。与基线文档「无重依赖」口径一致。

> ⚠️ 现状提示：依项目记忆与基线 §13，法术在 engine 侧未真正实装、前端存在部分死代码。本方案只动**渲染层与舞台容器**，不依赖法术实装；法术面板的死活不影响棋盘美化推进。

## 5. 美术方向（美工方案）

### 5.1 主题与色调

- 暗黑奇幻 / 巫师烛光夜色：深石板底 + 暖火光高光。
- 主色板：`#1a1d24`（舞台底）、`#3b3f4a`（石质塔身）、`#d9c9a3`（木/火光平台）、`#7fb3ff`（纹章蓝光）、`#e8b339`（选中金光）。
- 玩家色沿用 [config.ts](../../packages/client/src/game/config.ts) `PLAYER_COLORS`（红/蓝/绿/黄/紫/橙）。

### 5.2 塔（Tower3D）

- 石灰棱台，顶面带城齿（crenellation）凹槽贴图/边线，显「可站立平台」语义。
- 纹章塔：塔身侧面浮雕盾纹 + `#7fb3ff` 符文微光描边。
- 载城堡塔：顶面之上立小型尖塔/城堡件（替代现有 🏰 emoji）。
- 堆叠层：每段微调色阶使分层可读；**选中切片**（`onTowerClick` / `sliceStart`）整体镀金边 + 上抬 0.3 su + 阴影加深，明示「这段会一起移动」。
- 封印塔（`sealed`）：侧面加锁链贴图。

### 5.3 巫师（Wizard3D）

- 尖帽锥 + 袍身，按 `ownerPlayerId` 着色；无脸剪影。
- 状态：
  - 普通：立于塔顶平台或地面，按 3×2 排布自动落位。
  - 选中/可点：金光环 + 微浮动（复用现有 `highlight`）。
  - 封印（IMPRISONED）：按 §23.1 **不显示**；仅 debug 模式在塔体内显示半透明幽灵。
  - 不可点（F2 颜色隔离，`selectableWizardOwnerId` 不符）：降饱和 + 禁用光标。

### 5.4 空间格（Space3D）

- 石板方砖，纹章地面位（`isRavenShieldGround`）刻蓝色微光符文。
- 城堡在地面（`castleHere`）时显示城堡模型于格中心。
- 高亮：`target` 金描边、`selectable` 绿虚线，沿用现有语义、改用 3D 地面发光环。

### 5.5 相机

- 固定斜俯（`rotateX ≈ 55°`，无自由轨道）——保证 16 格环形始终可辨向。
- 仅缩放 + 平移，不做轨道旋转（避免玩家失向）。

## 6. 舞台容器规格（缩放/平移/还原）

容器替换 [GameContainer.tsx](../../packages/client/src/containers/GameContainer.tsx) 中 `main` 区域的 `<Board>` 直挂点。

```
<div class="stage-viewport" position:relative; overflow:hidden>   // 视口，剪裁溢出
  <div class="stage-world" transform: translate(panX,panY) scale(zoom)>  // 世界
    <Board3D ... />                                                // 3D 棋盘
  </div>
  <button class="stage-reset">🏚 一键还原</button>
</div>
```

| 行为 | 规格 |
|---|---|
| 默认缩放 | fit-all：`zoom = min(vw, vh) / (50 su · 18px)` 使舞台外接恰好入视口 |
| 滚轮缩放 | 以光标为锚点缩放（调整 pan 使锚点屏幕位不变）；范围 **0.5×–4×**；步长 ×1.1 |
| 拖放平移 | 背景mousedown→mousemove 改 pan；交互元素 `stopPropagation` 避免误触发 |
| 一键还原 | `zoom=fit-all, pan=(0,0)` |
| 触屏（后续） | 双指捏合缩放 + 单指拖动 |
| 锚定 | `transform-origin: center` |

**默认策略**：进入对局为 fit-all 全景；玩家滚轮放大交互，一键还原回全景。fit-all 下巫师 18 px 可见，2× 下 36 px 舒适可点。

## 8. 风险与备注

- **DOM 节点量**：极端全堆场景数百节点，React 渲染尚可；若掉帧，U5 阶段对非视口内格做 `content-visibility: auto`。
- **极端高堆遮邻格**：见 §3.5，靠缩放/平移补救，不做强制裁剪。
- **交互保真**：3D 化不得改变 `onWizardClick/onTowerClick/onSpaceClick` 的语义与触发条件（F2 颜色隔离、切片起点等）。
- **法术未实装**：本方案不依赖法术；若后续法术上线需 VFX，再评估迁方案 B。
- **回归**：Phase F 验收的 F1 解封覆盖 / F2 颜色隔离 / F3 弃 3 重抽 UI 必须在 U2–U4 后回归通过。

## 10. UI 布局与交互增强（本轮补充，2026-06-28）

> 本节为本次新增需求，与 §3–§7 的 3D 化并行。布局重构在 **U1** 阶段一并落地（舞台容器化时即切换整体布局）。

### 10.1 屏幕适配与主舞台比例锁定

- 目标比例：常规 **4:3 与 16:9** 均需适配（含其间任意比例）。
- 整体布局改为 **左舞台 / 右侧栏** 两栏：
  - 左侧 = 游戏舞台区（地图+塔+巫师），**比例固定不变**，且必须**填满**其所在栏的空间（按 §6 的 fit-all 缩放保证舞台外接恰好充满）。
  - 右侧栏 = 玩家/手牌/药水/法术（见 §10.3），宽度固定，纵向自适应。
- 舞台栏用 `aspect-ratio` 锁定世界比例（舞台外接 ≈ 1:1，见 §3.2），`overflow:hidden`；fit-all 缩放保证内容不变形地填满。
- 窄屏（< 1024px）降级：右侧栏折叠为底部抽屉（U5 处理）。

### 10.2 玩家轮次发光外框（双层）

- **页面外框 + 舞台外框** 双层发光，颜色 = 当前玩家色（`PLAYER_COLORS[current]`）。
- 实现：`body`/根容器 `box-shadow: inset` 外发光 + 舞台 viewport 外发光，颜色随 `state.currentPlayerId` 切换，过渡动画 200ms。
- 终局（`GAME_FINISHED`）关闭发光，改由 WinnerPanel 表达。
- 颜色映射沿用 [config.ts](../../packages/client/src/game/config.ts) `PLAYER_COLORS`。

### 10.3 右侧栏结构（自上而下）

```
┌─ 右侧栏（固定宽，纵向自适应） ─────────────┐
│ 1. 玩家信息区                              │
│    ┌────────────────────────────────────┐  │
│    │ 横向条：最多容纳 6 人，占满宽度     │  │
│    │  默认：主要信息（色标+计数）        │  │
│    │  hover：弹出该玩家详情              │  │
│    │  当前玩家：大完整面板（高亮其色）   │  │
│    │  其他玩家：小图标 + 主要信息        │  │
│    └────────────────────────────────────┘  │
│ 2. 手牌显示 + 手牌操作区                   │
│ 3. 魔法药水区：所有药水瓶状态              │
│    （空瓶 / 满药水 / 已使用）              │
│ 4. 法术区：法术用途 + 消耗药水数           │
└────────────────────────────────────────────┘
```

**1. 玩家信息区**：
- 顶部横向条，**预留可容纳最大玩家数（6）的等宽槽位**，无论实际人数都占满宽度；高度尽量小。
- 默认每格显示主要信息（玩家色标 + 巫师/药水计数等关键指标）。
- **鼠标悬浮**任意格 → 弹出该玩家详情浮层。
- **当前轮次玩家**：呈现大的完整信息面板（展开式，占据更多纵向空间）；其余玩家保持小图标 + 主要信息。
- 当前玩家高亮其颜色，与 §10.2 发光呼应。

**2. 手牌区**：手牌展示 + 出牌操作（沿用现有 [HandPanel.tsx](../../packages/client/src/components/HandPanel.tsx) 交互，重排到右侧）。

**3. 魔法药水区**：**仅显示当前操作玩家**的所有魔法瓶实际状态——空瓶 / 满药水 / 已使用三态。需新建组件读取 `state.players[current].potions`（或对应字段，实装时核对数据字典）。其他玩家的药水详情**不展示**（未轮到时不可见）。

**4. 法术区**：每个可用法术显示**用途说明 + 所需药水数**；现有 [SpellPanel.tsx](../../packages/client/src/components/SpellPanel.tsx) 需扩展「消耗药水数」字段（来自法术定义）。

### 10.3.1 信息可见性

- 玩家信息条（§10.3 第 1 区）显示**公开信息**（色标 + 公开计数，如剩余巫师数等），所有玩家可见。
- **私密信息**（手牌内容、药水详情、详情悬浮浮层里的非公开字段）**仅当前操作玩家可见**；未轮到的玩家不展示他人私密信息。
- 药水区（第 3 区）只渲染当前玩家，是其具体体现。

### 10.6 日志面板（LogPanel）

- 现有 [LogPanel.tsx](../../packages/client/src/components/LogPanel.tsx) 从底部栏移出，改为**角落图标**（如右下角 📜）。

### 10.4 卡牌选中后的巫师高亮（核心交互增强）

**需求**：玩家点击卡牌、选择塔或巫师后，把「能够操作的巫师」高亮（外框发光背景效果）；**塔不施加高亮效果**。

**不做可达性预演**：本轮**不预测/不预演**移动是否合法——不判定目标落点是否可达、是否超容量、切片是否可行等。高亮只表达「这是你当前这一步可操作的对象集合」，不表达「它一定移得动」。点击后仍由引擎校验，非法时按现有报错路径（`error` 提示）处理。

**高亮规则（按 intent 状态）**：
| intent 状态 | 高亮对象 | 说明 |
|---|---|---|
| `PLAY_CARD_WIZARD`（巫师牌 / 二选一选巫师） | 当前玩家所有可见巫师 | 叠加 F2 颜色隔离：仅 `owner === current` |
| `CAST_SPELL_MOVE_WIZARD` | 可被该法术操作的巫师集合 | 法术移动巫师选源 |
| `CAST_SPELL_FREE_WIZARD` | 被封印的巫师 | 仅 debug 模式可见；正常对局按 §23.1 不揭示，此项仅 debug 生效 |
| `PLAY_CARD_TOWER_PICK` / `CAST_SPELL_MOVE_TOWER` / `DISCARD_REDRAW_FLOW`（切片/塔相关） | **无高亮** | 塔保持常规外观，可点但无发光 |
| `IDLE` / 其余 | 无高亮 | — |

**视觉效果**：巫师外框发光背景（玩家色环或金色光环，复用既有 `highlight` 语义升级为发光）；**塔一律不加发光类**，即便处于塔相关意图中。

**实现**：在 `intent` 进入相应状态时，于 [GameContainer.tsx](../../packages/client/src/containers/GameContainer.tsx) 计算 `highlightableWizards: Set<WizardID>`，传入 Board3D / Wizard3D；集合内巫师加发光，集合外无发光。**无需 engine 侧改动**（不预演合法性），仅在 client 侧按 intent + F2 颜色隔离过滤。

**与 F2 颜色隔离的关系**：F2（`selectableWizardOwnerId`）限制「只能点自己可见巫师」；本高亮在此基础上标识「当前可操作」。两者同源：高亮集合 = F2 过滤后的可见巫师（按 intent 范围收窄）。

### 10.5 与既有交互语义的对齐

- 切片选择（`sliceStart` / `onTowerClick`）、城堡/纹章/封印标记、封印巫师不揭示（§23.1）等既有语义**全部保留**。
- 高亮三态语义不变：`target`（金）、`selectable`（绿虚线），3D 化后改为地面发光环 + 选中上抬。
- 出牌/施法/弃 3 重抽（F3）点击链路不破，U2–U4 后回归 Phase F 的 F1/F2/F3。
