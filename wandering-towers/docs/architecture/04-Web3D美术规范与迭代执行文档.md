## 1. 背景与目标

当前参考页 `WizardTower_Stage_Reference_FIXED.html` 已验证了最基础的 3D 舞台能力：可旋转、可放大缩小、可摆放塔和巫师。但它只是粗糙模型：塔体偏简单，乌鸦堡只是黑色方块，地图与道具仍然缺少统一的 3D 游戏风格。

项目正式代码位于：

- 前端：`packages/client/src`
- 当前 3D 相关组件：`Board.tsx`、`Space.tsx`、`StageViewport.tsx`、`Tower3D.tsx`、`Wizard3D.tsx`、`Castle3D.tsx`
- 样式：`packages/client/src/styles/phase-u.css`

本轮迭代目标不是做展示页，而是把真实游戏界面中的地图、塔、乌鸦堡、巫师、药水、卡牌和交互状态统一升级为可玩的 3D 美术系统。

必须满足：

1. 塔可以纵向叠放，极端情况下 9 层塔不能出画面。
2. 前景塔尽量不要遮挡后方塔，默认视角下 16 个空间都应可读。
3. 每个空间的地面或塔顶最多可见 6 个巫师，塔顶平台必须按 6 个巫师反推尺寸。
4. 乌鸦堡不是装饰方块，而是可叠放在塔顶或地面的独立 3D 模型。
5. Web3D 美术迭代不能破坏现有游戏规则、点击链路和状态语义。

## 2. 技术路线

### 2.1 当前阶段采用 CSS 3D，不引入 Three.js

正式项目当前是 React + Vite + CSS 3D，不依赖 three.js。后续 agent 应优先沿用当前架构：

- `StageViewport` 负责舞台视口、缩放、平移、固定俯视角。
- `Board` 负责 16 个空间的环形布局。
- `SpaceCell` 负责单个空间中的地面、塔堆、巫师、乌鸦堡。
- `Tower3D`、`Wizard3D`、`Castle3D` 分别负责 3D 资产组件。
- `phase-u.css` 承载全局舞台、光效、动画、浮层样式。

不要在本轮直接把主游戏改成 Three.js / react-three-fiber，除非用户明确要求。原因是当前游戏交互依赖 DOM 点击，CSS 3D 可以保留已有 `onSpaceClick`、`onWizardClick`、`onTowerClick`，不需要重建 raycast 交互层。

### 2.2 参考页的定位

`WizardTower_Stage_Reference_FIXED.html` 只作为“3D 舞台方向参考”，不能直接照搬：

- 可以参考 OrbitControls 原型中的旋转、缩放体验。
- 正式游戏默认不开放自由旋转，避免玩家迷失方向。
- 正式游戏使用固定斜俯视角 + 缩放 + 平移。
- 正式美术必须围绕真实规则：16 格环形地图、9 座塔、塔可堆叠、6 巫师容量、乌鸦堡可上塔。

## 3. 核心比例规范

统一使用一套逻辑单位，后续可映射为 CSS px。

### 3.1 基础约束

| 项目 | 规范 |
|---|---|
| 地图空间数 | 16 |
| 塔总数 | 9 |
| 单空间可见巫师上限 | 6 |
| 巫师排列 | 3 列 x 2 行 |
| 最高塔堆 | 9 层 |
| 乌鸦堡位置 | 可在地面，也可在某座塔顶 |
| 默认视角 | 固定斜俯视，不自由旋转 |

### 3.2 以巫师反推塔顶平台

塔顶平台必须能容纳 6 个巫师。建议逻辑尺寸：

- 巫师底座直径：`1.0 su`
- 巫师间距：`0.3 su`
- 3 列宽度：`3 * 1.0 + 2 * 0.3 = 3.6 su`
- 2 行深度：`2 * 1.0 + 1 * 0.3 = 2.3 su`
- 塔顶平台最终取整：`4.0 su x 4.0 su`

结论：塔顶不能做得太小。当前 `Tower3D` 约 60px 平台、`Wizard3D` 约 14px 巫师是可玩原型，但美术迭代时要建立明确比例，保证 6 个巫师站满时仍然不挤、不穿模。

### 3.3 推荐 CSS 像素映射

建议先用以下基准重构：

| 元素 | 逻辑尺寸 | 建议 CSS 尺寸 |
|---|---:|---:|
| 巫师底座 | 1.0 su | 16px |
| 巫师总高 | 1.4 su | 22px |
| 塔顶平台 | 4.0 su | 72px |
| 单层塔高 | 2.0 su | 36px |
| 空间地块 | 6.0 su | 108px |
| 环形格距 | 7.2 su | 130px |
| 舞台世界边长 | 约 52 su | 936px |

如需更保守地保证 9 层不出画面，可使用：

- 塔顶平台：`68px`
- 单层塔高：`32px`
- 空间地块：`104px`
- 舞台世界边长：`900px`

### 3.4 9 层塔不出画面

9 层塔堆高度推荐：

- 单层塔 CSS 高度：`32px - 36px`
- 9 层总视觉高度：`288px - 324px`
- 默认 fit-all 缩放下，最高塔必须完整留在舞台视口内。

实现要求：

1. `StageViewport` 的 fit-all 计算不能只按平面地图宽高算，还要预留最高塔向上投影空间。
2. 舞台世界边界要包含顶部安全边距，建议顶部额外预留 `180px - 240px`。
3. 当视口较小时，默认 zoom 应自动降低，保证 9 层塔不会被裁切。
4. 允许玩家放大查看细节，但默认视图必须完整。

## 4. 视角与遮挡规范

### 4.1 默认相机

采用固定斜俯视角：

- `rotateX(50deg - 58deg)`，建议初始 `55deg`
- 不开放自由旋转
- 允许鼠标滚轮缩放
- 允许拖拽平移
- 提供一键还原 fit-all

### 4.2 减少前景遮挡后景

地图是 16 格环形，如果直接用纯正侧视，高塔会挡住后方。后续 agent 应做以下处理：

1. 环形地图整体保持近似正圆或轻微椭圆，不要压成横条。
2. 塔体使用“矮胖平台塔”，不要做成细高尖塔。
3. 单层塔高不超过塔顶宽度的 `0.5 - 0.55`。
4. 塔堆使用轻微层间错色和阴影，不用过大的屋顶或尖顶。
5. 前景空间可以轻微降低视觉透明遮挡风险：高塔侧面阴影不要全黑，避免压住后方目标。
6. 选中、可点击、目标状态必须用发光轮廓表达，不能只靠被遮挡的小文本。

### 4.3 层级排序

DOM 排序必须匹配视觉前后关系：

- 后方空间先渲染，前方空间后渲染。
- 或者根据空间的 y 坐标动态设置 `z-index`。
- 同一空间内部：地面 < 塔堆 < 塔顶巫师 / 乌鸦堡 < 选中光效。

当前 `Board.tsx` 只是按数组顺序渲染，后续应改为根据环形坐标的屏幕 y 值排序，避免后方塔错误盖住前方塔。

## 5. 地图空间 Space3D 规范

每个空间不是普通 2D 方块，而是一个可站立、可点击、可承载塔堆的 3D 地块。

### 5.1 外观

建议风格：

- 暗色奇幻石板地面。
- 每格是厚石台或浮空石砖，不是白色卡片。
- 格子之间通过环形道路连接，形成“飞塔环形轨道”。
- 乌鸦纹章位置使用冷蓝符文。
- 起始容量位置使用小型火焰或蜡烛槽，但不要使用 emoji。

### 5.2 状态

| 状态 | 表现 |
|---|---|
| 普通 | 暗石板、微弱边缘高光 |
| 可选择 | 绿色或青色发光边框 |
| 目标 | 金色符文环或金色描边 |
| 乌鸦纹章地面 | 蓝色符文刻印 |
| 有地面城堡 | 地面中心摆放 Castle3D |
| 有地面巫师 | 按 3 x 2 阵列站在地面平台前缘或中心 |

### 5.3 尺寸

单格必须同时容纳：

- 最多 6 个地面巫师。
- 1 个塔堆底座。
- 地面状态符文。
- 空间编号或 debug 信息。

正式美术中空间编号应缩小为调试标记，不作为主要视觉。

## 6. 塔 Tower3D 规范

### 6.1 塔的核心语义

塔不是装饰物，而是游戏中的可移动堆叠单位。每一层塔都必须清楚表达：

- 这是第几层。
- 它是否是乌鸦纹章塔。
- 它是否在当前被选择的切片中。
- 它是否可点击作为切片起点。
- 它顶部是否承载乌鸦堡或巫师。

### 6.2 外观方向

建议做成“可站人的魔法石塔平台”：

- 方形或八边形石台。
- 顶部是明确平台，不做尖屋顶。
- 边缘有城垛、石砖、符文槽。
- 侧面有石砖缝、裂痕、轻微色块，不要纯色矩形。
- 乌鸦纹章塔侧面嵌入蓝色符文盾牌图案。

禁止：

- 把塔做成细尖塔，导致上面站不了巫师。
- 用 emoji 表示核心资产。
- 乌鸦堡继续使用黑色方块。

### 6.3 堆叠方式

塔堆应使用稳定的垂直布局：

- 底层在下，上层在上。
- 每层高度固定。
- 层与层之间可以有 1px - 2px 的视觉分隔。
- 上层比下层略微亮一点，帮助读层。
- 不要让每层随机偏移，避免玩家误解塔堆结构。

### 6.4 选中切片表现

当玩家点击某层塔作为移动起点时，该层及其上方所有塔属于同一移动切片。

表现要求：

- 切片整体金色描边或金色外发光。
- 切片可轻微上抬 `4px - 6px`。
- 只高亮切片，不高亮整格地面。
- 巫师选择状态下，不给塔添加发光，避免误导。

## 7. 乌鸦堡 Castle3D 规范

乌鸦堡必须是独立 3D 模型，可在地面或塔顶显示。

### 7.1 尺寸

乌鸦堡要能站在塔顶平台上，但不能挤掉巫师的可读性：

- 宽度：塔顶平台的 `55% - 65%`
- 高度：单层塔高的 `1.1 - 1.3` 倍
- 底部必须有小阴影，表示它站在平台上

如果乌鸦堡与巫师同时在同一塔顶，需要优先保证游戏语义可读：

- 乌鸦堡居中靠后。
- 巫师 3 x 2 阵列居中靠前或围绕堡底。
- 不允许乌鸦堡完全挡住巫师。

### 7.2 外观

风格关键词：

- 暗石城堡
- 乌鸦尖塔
- 金色小窗
- 暗红屋顶或黑蓝屋顶
- 旗帜或乌鸦剪影

不要再用黑色方块。当前 `Castle3D.tsx` 已经是较好的起点，但后续应：

- 去掉乱码 title。
- 避免 emoji。
- 增加更清楚的乌鸦主题剪影。
- 让它可以在地面和塔顶共用同一套组件。

## 8. 巫师 Wizard3D 规范

### 8.1 外观

巫师是可点击棋子，造型应简洁但可读：

- 尖帽
- 袍身
- 小脸或面罩暗部
- 玩家颜色作为袍子主色
- 底部投影
- 选中时发光环

避免复杂五官，因为巫师在全景时很小。

### 8.2 站位

每个地面或塔顶最多 6 个巫师，固定使用 3 x 2 阵列。

建议站位：

```
后排：1 2 3
前排：4 5 6
```

在斜俯视角中，后排应略微上移，前排略微下移。不要让 6 个巫师挤成一条横线。

### 8.3 交互状态

| 状态 | 表现 |
|---|---|
| 普通 | 玩家颜色袍子 + 底部阴影 |
| 可点击 | 鼠标指针 + 轻微亮边 |
| 高亮 | 金色或玩家色光环，轻微浮动 |
| 不可点击 | 降饱和、透明度降低 |
| 被封印 | 正常对局不显示；debug 模式才可半透明显示在塔内 |

高亮规则必须保持：选中巫师牌或巫师相关法术时，高亮可操作巫师；塔相关意图不高亮巫师。

## 9. 舞台 StageViewport 规范

### 9.1 功能

舞台必须支持：

- 默认 fit-all，完整显示地图和最高塔安全边界。
- 鼠标滚轮缩放。
- 拖拽平移。
- 一键还原。
- 不破坏点击塔、巫师、空间的交互。

### 9.2 默认缩放

当前 `StageViewport.tsx` 使用 `WORLD_SIZE = 620`。后续如果按新比例重构，必须同步更新：

- `WORLD_SIZE`
- `Board.tsx` 中的环半径、格子尺寸
- CSS 中舞台背景环形道路尺寸
- fit-all 计算中的最高塔安全边距

建议新世界尺寸从 `900px - 980px` 起步，不要继续沿用 620px 的小世界，否则塔顶 6 巫师和 9 层塔很难兼顾。

### 9.3 视口裁切

舞台 `overflow: hidden` 可以保留，但默认视图不能裁切核心内容。

验收方法：

1. 构造 9 层塔堆。
2. 把乌鸦堡放在最高塔顶。
3. 塔顶放满 6 个巫师。
4. 在 1366 x 768、1920 x 1080、1024 x 768 三种窗口下默认进入游戏。
5. 最高点不能被裁掉。

## 10. 道具与侧边 UI 的 3D 化

本轮重点是地图与道具都按 3D 样式呈现，不只做棋盘。

### 10.1 卡牌

卡牌应保留当前交互，但升级视觉：

- 使用实体卡牌样式：厚度、阴影、轻微倾斜。
- 选中卡牌上浮并发光。
- 移动巫师牌、移动塔牌、二选一牌使用不同图标或角标。
- 不要用大段说明文字占据卡面。

### 10.2 药水

药水区域应显示为小瓶道具：

- 空瓶：透明玻璃、灰色液位。
- 满瓶：彩色液体、微光。
- 已使用：暗淡、瓶塞打开或裂纹。

药水只显示当前操作玩家的私密状态，不展示其他玩家私密信息。

### 10.3 法术

法术按钮应更像可施放的魔法符文槽：

- 显示法术用途。
- 显示消耗药水数。
- 可用时发光。
- 不可用时降饱和。

### 10.4 日志

日志保持角落图标 + 展开浮层，不占主舞台空间。

涉及文件：

验收：

目标：

- 保持右侧栏结构和信息权限。

验收：

- 卡牌选中、药水状态、法术可用状态清晰。
- 不泄露非当前玩家私密信息。
- 窄屏布局仍可用。

## 12. 验收清单

每个阶段完成后，后续 agent 必须执行以下检查：

1. 运行 `pnpm --filter @wt/client typecheck`。
2. 如改动共享类型或规则相关代码，运行 `pnpm test`。
3. 手动启动前端并检查游戏首屏。
4. 检查巫师牌、塔牌、法术点击链路。
5. 检查缩放、平移、一键还原。
6. 检查 1366 x 768、1920 x 1080、1024 x 768 三个视口。
7. 构造或模拟 9 层塔 + 乌鸦堡 + 6 巫师的极端场景。
8. 检查前景塔是否明显遮挡后方关键对象。
9. 检查核心资产是否仍有 emoji 或黑色占位块。
10. 检查是否出现中文乱码。

## 13. 不应做的事

后续 agent 不要：

1. 不要重写规则引擎。
2. 不要把主游戏直接改成自由 OrbitControls。
3. 不要为了 3D 效果牺牲点击交互。
4. 不要使用黑色方块、emoji 或纯色矩形作为最终核心资产。
5. 不要让塔顶平台小到无法站 6 个巫师。
6. 不要让 9 层塔默认出画面。
7. 不要在 UI 中展示其他玩家的私密手牌或药水详情。
8. 不要只做静态展示图，必须服务真实游戏状态。
