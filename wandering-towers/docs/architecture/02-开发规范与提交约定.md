# 开发规范与提交约定

> 版本：1.0　|　创建：2026-06-25　|　关联：[00-开发任务总表](../00-开发任务总表.md)

## 1. 分支与提交
- 主分支 `main`；功能分支 `feat/phaseN-<task>`，如 `feat/phase1-move-wizard`。
- 提交信息 Conventional Commits：`feat(engine): 实现精确巫师移动`、`fix(engine): 修正封印归属`、`test: 补 TC-WIZ-005`、`docs: 补 Phase1 日志`。
- 每个任务一个 PR/合并点；合并前 `pnpm -r typecheck && pnpm test` 须通过。

## 2. 代码规范
- TypeScript strict；禁 `any`，必要时 `unknown` + 守卫。
- 枚举用字符串字面量联合类型（对齐 V3 数据字典）。
- 引擎函数纯函数化，副作用仅经事件流。
- 禁止把状态藏在前端（V4 §1.3）。
- 文件名 kebab-case；导出用具名导出。

## 3. 命名口径（V4 §11.3、V2 §26）
- 封印 / 解封（不用「囚禁/释放」作内部术语，但枚举值保留 `IMPRISONED` 以兼容数据字典）。
- 乌鸦城堡 / RavenCastle；药水 Potion（EMPTY/FULL/SPENT）。

## 4. 规则口径红线（来自 V4 §11、V2 §23、V2 开发需求 §50）
1. 开局容量走 setupCapacity（火苗数），不写死 3/2/1。
2. 药水三态 EMPTY→FULL→SPENT，单向不可逆。
3. 终局药水条件 = EMPTY==0，非「桌面全满」。
4. 封印多巫师只翻 1 瓶；封自己也可翻。
5. 巫师进堡立即结束回合（含法术致进堡）。
6. 塔不能压堡，但堡可随所在塔移动。
7. 法术系统可配置，不写死 2 张；走三层架构。
8. 进堡必须精确步数，过堡不进，有余步越过。

## 5. 测试规范
- 每个 TC-* 用例对应一个测试文件或 `describe` 块，标题含用例编号。
- 引擎测试不依赖前端；用 fixtures 工厂构造 GameState。
- 不变量检查器（V3 §23）作为每个集成测试的收尾断言。
- 回放测试：录一局 → 回放 → 深比较最终 GameState。

## 6. 文档与日志规范
- 每完成一个任务：在对应 Phase 开发日志追加一条记录（见 [dev-log/README.md](../dev-log/README.md) 模板）。
- 遇到规则歧义：在日志记「规则口径裁定」并引用 V2/V4 出处。
- 重大决策记入架构文档 §8 决策记录表。

## 7. 跨平台检查清单（每次提交前）
- [ ] 无 shell 专有脚本（用 cross-env）
- [ ] 路径用 `path.join`
- [ ] 无新增原生依赖
- [ ] 文件名小写 kebab-case
- [ ] `.gitattributes` 已强制 lf
