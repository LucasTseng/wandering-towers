## 1. 项目目标与边界

### 1.1 目标（V4 §1.1）
开发《巫师飞塔》**标准版（基础游戏）**数字化可玩版本：
1. 2–6 人对局
2. 完整基础规则
3. 完整标准版法术系统（7 法术，三种启用模式）
4. 本地单机/本地托管（hot-seat，单用户控所有玩家）
5. 规则引擎 + 前端棋盘 + 基础回放
6. 为联机与扩展包预留数据结构

### 1.2 边界（V4 §1.2）
- **实现**：地图/空间、塔堆、巫师四态、乌鸦城堡、移动牌、弃3重抽、封印/解封、药水三态、法术、终局、回放。
- **不实现**：扩展包、排位/账号、社交、高级动画、AI 强策略。
- **保留扩展能力**：法术走「定义+校验+执行」三层；地图数据化 MapDefinition；事件流驱动；显式状态模型。

## 5. 数据模型（V3 数据字典 §1-14，已对齐代码）

> 完整定义见 [packages/shared/src/types/](../../packages/shared/src/types/)。以下为关键结构摘要，字段语义以代码为准。

### 5.1 ID 规范（V3 §2）
`PlayerID=P1..`、`WizardID=W_<P>_<NN>`、`TowerID=T01..T09`、`PotionID=PT_<P>_<NN>`、`CardID=C_NNNNN`、`SpellID` 固定。

### 5.2 枚举（V3 §3，已对齐）
- **GameMode**: BASIC / CUSTOM / MASTER_VARIANT
- **TurnPhase**: TURN_START → ACTION_1 → ACTION_2 → TURN_END → GAME_END_PENDING → GAME_FINISHED
- **WizardStateType**: ON_GROUND / ON_TOWER_TOP / IMPRISONED / IN_CASTLE
- **PotionState**: EMPTY → FULL → SPENT（单向不可逆）
- **MovementCardType**: MOVE_WIZARD / MOVE_TOWER / MOVE_WIZARD_OR_TOWER
- **ActionSource**: MOVEMENT_CARD / SPELL / DISCARD_REDRAW_TOWER_BONUS
- **CommandType**: PLAY_CARD / DISCARD_REDRAW / CAST_SPELL / END_TURN / CHOOSE_DICE_RESULT / SKIP_SECOND_ACTION
- **EventType**: 见 §8
- **RuleErrorCode**: 见 §9.3

### 5.3 核心状态结构

**GameConfig**（`types/game-config.ts`）：
```ts
{ playerCount: 2..6, mode: GameMode, spellSetup: SpellSetupConfig, seed?: number }
// SpellSetupConfig: { mode, selectedSpellIds?, spellPoolSource?, spellSelectionMode?,
//   spellCount?, maxSpellsPerTurn, nonActivePlayersCanCast, castTimingMode }
```

**GameState**（`types/game-state.ts`）顶层字段：`config, currentPlayerId, turnPhase, roundNumber, board, towers, wizards, potions, players, ravenCastle, drawPile, discardPile, availableSpells, endgameTriggered, endgameTriggerPlayerId, endgameTriggerRound, winners, scores, sharedVictory, stateVersion, playerOrder`。

> ⚠️ **实现细节**：state-builder 把 `cardId→templateId` 映射挂在 `(state as any)._cardTemplates`（非序列化字段），供 play-card 反查牌面。这是当前实现的妥协点，未来若做存档/联机需正式化为 GameState 字段或独立 lookup。

**WizardState**（`types/wizard.ts`，判别联合）：
```ts
| { mode:'ON_GROUND'; spaceIndex }
| { mode:'ON_TOWER_TOP'; spaceIndex; topTowerId }
| { mode:'IMPRISONED'; spaceIndex; insideTowerId }
| { mode:'IN_CASTLE' }
```

**SpaceState**（`types/board.ts`）：`{ index, groundHasRavenShield, setupCapacity, groundVisibleWizards[], towerStack[] }`，towerStack 自下而上。BoardState 固定 16 空间。

**TowerRuntimeState**（`types/tower.ts`）：`{ id, hasRavenShield, imprisonedWizards[], sealed }`。注意：塔的空间位置/层级由 `board.spaces[x].towerStack` 决定，TowerRuntimeState 不存所在空间（V2 §6.1）。`sealed` 是 T1.6 扩展的「塔封印锁定」机制，与巫师 IMPRISONED 是两个概念。

**Potion**（`types/potion.ts`）：`{ id, ownerPlayerId, state }`。

**RavenCastleState**（`types/raven-castle.ts`）：`{ position: ON_SPACE|ON_TOWER, wizardIdsInside[] }`。

**PlayerState**（`types/player.ts`）：`{ id, seatIndex, color, wizardIds[], potionIds[], hand[], spellsCastThisTurn }`。

### 5.4 静态定义层
- **STANDARD_MAP**（`map-definitions/standard.ts`）：16 空间；塔位 space 1–9，setupCapacity = [3,3,3,2,2,2,1,1,1]；5 座纹章塔（T01–T05）+ 4 普通（T06–T09）；城堡初始 space 0（地面纹章位）。
- **PLAYER_RESOURCES**（`constants/player-resources.ts`）：2人=5巫师/6药水 … 6人=3巫师/4药水。
- **MOVEMENT_CARD_TEMPLATES** + TEMPLATE_QUANTITIES（`movement-card-definitions/standard-deck.ts`）：18 模板，合计 90 张。**注意**：文档未给 90 张精确分布，当前为合理推断，可后续据官方牌表校准（`assertDeckTotal()` 守护合计）。
- **STANDARD_SPELL_POOL**（`spell-definitions/standard-pool.ts`）：7 法术，见 §10。

## 6. 引擎 API 契约（真实签名）

### 6.1 入口 `RuleEngine`（`rule-engine.ts`）
```ts
class RuleEngine {
  constructor(config: GameConfig, options?: { seed?: number });
  // 构造时内部 buildInitialState，state/rng/initEvents 就绪
  readonly state: GameState;
  readonly rng: Rng;
  execute(command: ActionCommand): ExecuteResult;
  getInitEvents(): GameEvent[];
  getAllEvents(): GameEvent[];   // 当前等同 initEvents（命令事件未单独累计）
}
interface ExecuteResult {
  success: true; events: GameEvent[]; endTurn: boolean;
  endgameTriggered: boolean; stateVersion: number;
}
```
> ⚠️ **与 V3 §21 的差异**：V3 建议引擎返回 `{success, events, endTurn, endgameTriggered}`（不含 stateVersion），当前实现额外返回 `stateVersion`，并在 `execute` 内部直接 `applyEvents` 到 `this.state`。即引擎**既产事件又改状态**（非纯函数返回）。这是与架构文档「execute 为纯函数」的偏差点，见 §13 待办。

### 6.2 状态构建 `buildInitialState`（`state/state-builder.ts`）
```ts
function buildInitialState(config: GameConfig, rngSeed?: number): {
  state: GameState; events: GameEvent[]; rng: Rng;
}
```
按 V4 §14.2 顺序：校验配置 → 建棋盘 → 建玩家/巫师/药水 → 建9塔 → 建洗牌牌堆 → 解析法术池 → 摆城堡 → 摆塔 → 每人发3牌 → 按 setupCapacity 摆巫师到塔顶（玩家轮流） → 设 currentPlayer/phase/round → emit INIT_COMPLETED。

法术池解析 `resolveAvailableSpells`：
- BASIC → `[MOVE_WIZARD_1, MOVE_TOWER_2]`
- CUSTOM/MASTER + FIXED → `selectedSpellIds`（校验合法 + 模式允许）
- CUSTOM/MASTER + RANDOM → 从合法池 `rng.shuffle` 取前 `spellCount` 张

### 6.3 选择器 `selectors.ts`（纯查询，不修改状态）
```
towerStackAt, topTowerAt, groundWizardsAt, towerTopWizardsAt,
visibleWizardsAt, visibleWizardCountAt, imprisonedWizardsInTower,
towerSpaceIndex, isWizardOnTower, isWizardVisible, wizardsOfPlayer,
potionsByState, countPotions, allWizardsInCastle,
ravenCastleSpaceIndex, ravenCastleOnTowerId, clockwiseSpace, isRavenShieldPosition
```
> ⚠️ **重要勘误**：历史死代码（cast-spell.ts / spells/** / 两个 disabled spec）使用的 `getPlayer` / `getWizard` / `getTower` / `getPotionsByState` / `getPotionsByPlayer` / `getWizardsAt` / `getTowerStack` **均不存在**。真实选择器是上表命名。

### 6.4 事件应用 `apply-event.ts`
```ts
function applyEvent(state: GameState, event: GameEvent): void;   // 唯一状态变更入口
function applyEvents(state: GameState, events: GameEvent[]): void;
```
> ⚠️ **实现细节**：`TOWER_SLICE_MOVED` 的 payload 实际带 `wizardsOnTop` 字段（不在 game-event.ts 的 `TowerSliceMovedPayload` 类型里，运行时补字段）。这是类型与实现的偏差，待统一。

### 6.5 随机 `rng.ts`
```ts
interface Rng { next(): number; nextInt(min,max): number; shuffle<T>(arr): T[]; readonly seed: number; }
function createRng(seed: number): Rng;
function rollDie(rng: Rng): number;   // 1..6
```

### 6.6 事件累加 `event-bus.ts`
```ts
class EventAccumulator {
  constructor(startSequence: number);
  push(type, payload, actorPlayerId?, gameId?): GameEvent;  // 自增 sequence、分配 eventId
  all(): GameEvent[]; count(): number; nextSequence(): number;
}
```
> ⚠️ **勘误**：死代码引用的 `EventBus` 接口（带 `emit`/`getEvents`/`getAccumulatedState`）**不存在**。真实只有 `EventAccumulator`。规则函数的 emit 签名是 `(type, payload) => GameEvent`（见 tests/helpers.ts `mkApplyEmit`）。

### 6.7 不变量 `invariants.ts`
```ts
function checkInvariants(state): InvariantViolation[];   // 五类：塔计数/唯一、封印登记一致、药水总数、城堡巫师一致
function assertInvariants(state): void;                   // 有违例则抛
```

### 6.8 回放 `replay/replay.ts`
```ts
function replay({ config, seed, events }): GameState;     // buildInitialState(config,seed) + applyEvents
function replayEmpty(config, seed?): GameState;
```

**emit 约定**：规则函数是事件溯源式——只 `emit(type, payload)` 产事件、不直接改 state；由调用方（RuleEngine.execute 或测试的 mkApplyEmit）决定何时 applyEvent。

### 7.1 `RuleEngine.execute` 命令分发（真实）
```
PLAY_CARD         → playMovementCard
DISCARD_REDRAW    → discardRedraw
SKIP_SECOND_ACTION→ skipSecondAction
END_TURN          → endTurn=true（触发 endTurnCleanup）
CAST_SPELL        → ✅ castSpell（spells/cast-spell.ts，Phase B 已接入）
CHOOSE_DICE_RESULT→ ❌ throw INVALID_PHASE "not implemented yet"（骰子牌独立任务）
```
结算后：`applyEvents(state, 命令事件)`；若 `endTurn` → `endTurnCleanup` + 若终局 → `finalizeWinners` + 置 `GAME_FINISHED`。
